## New observation (minor, informational only)

Splitting `App.tsx` introduced one small, low-severity duplication: five of the six view components independently repeat the same two-element wrapper (`<main className="app-shell"><section className="panel …-panel">…</section></main>`) — `GameView` is the exception, using `game-shell` instead. This is exactly the kind of trade-off that comes with extracting components: each file is now more readable in isolation, at the cost of ~3 lines of repeated markup per file.

This is **not** worth fixing proactively. A shared `<ViewShell>` wrapper would remove the duplication but would need a prop for the panel's modifier class (`help-panel`, `results-panel`, etc.) and would be the kind of abstraction introduced for five call sites doing something extremely simple — the "three similar lines is better than a premature abstraction" bar applies directly here. Flagging it so it's a documented, deliberate non-fix rather than an oversight; worth revisiting only if a 7th or 8th view is added and the pattern starts to feel load-bearing rather than incidental.

## Remaining known gaps (carried over, not fixed, and correctly out of scope)

These were named in the initial review as reasonable omissions for this project's scope, not defects. They still apply and are listed here for completeness rather than re-litigated:

- No React error boundary — a genuine runtime bug in game logic would still produce a blank white screen with no recovery UI.
- No enforced/measured test-coverage percentage (no coverage threshold wired into `vitest` or CI).
- `vite.config.ts` reads `package.json` via `node:fs.readFileSync` rather than a native JSON import, despite `resolveJsonModule: true` already being set in `tsconfig.app.json` — a working, low-risk choice, not idiomatic, not worth the churn to change.
